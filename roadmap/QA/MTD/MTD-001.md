# MTD-001 — Autenticação com Cookie HttpOnly

## Flow
[FLOW-001](../Flow/FLOW-001.md) — Autenticação e Sessão

## Objetivo
Validar que o login persiste sessão exclusivamente em cookie HttpOnly, que a sessão sobrevive ao restart do servidor, e que o logout remove completamente o cookie.

## Pré-condições
- Ambiente configurado com `JWT_RSA_PRIVATE_KEY` e `JWT_RSA_PUBLIC_KEY` no Railway (ou variáveis locais)
- Usuário cadastrado com e-mail e senha válidos
- Acesso ao DevTools do browser (Chrome/Firefox)
- Acesso ao servidor para executar restart (ou ambiente de staging)

---

## Cenário 1 — Login e validação de cookie HttpOnly

### Passos
1. Acessar a tela de login do frontend
2. Inserir credenciais válidas (e-mail + senha)
3. Clicar em "Entrar"
4. Após redirecionamento, abrir DevTools → aba **Application**
5. Navegar em **Cookies** → selecionar o domínio da aplicação
6. Localizar o cookie `accessToken`
7. Verificar as flags do cookie
8. Abrir aba **Local Storage** → confirmar que está vazio para o domínio
9. Abrir aba **Session Storage** → confirmar que está vazio para o domínio

### Resultado Esperado
- Cookie `accessToken` presente com flags: `HttpOnly ✓`, `Secure ✓`, `SameSite: Strict`
- **Zero** tokens armazenados em Local Storage ou Session Storage
- Usuário redirecionado para o dashboard após login bem-sucedido

### Evidência
- [ ] Screenshot da aba Application → Cookies mostrando `accessToken` com flags
- [ ] Screenshot da aba Application → Local Storage (vazia)

---

## Cenário 2 — Persistência de sessão após restart do servidor

### Passos
1. Realizar login com credenciais válidas
2. Navegar por pelo menos 3 páginas dentro do produto (ex: `/dashboard`, `/items`, `/maintenances`)
3. Anotar a URL atual
4. Reiniciar o servidor da API (simular deploy no Railway ou reiniciar localmente)
5. Aguardar o servidor subir novamente (verificar health check)
6. Voltar ao browser e recarregar a página atual (F5)
7. Verificar se o usuário ainda está autenticado

### Resultado Esperado
- Usuário permanece autenticado após o restart
- Nenhum redirecionamento para a tela de login
- Nenhuma mensagem de "sessão expirada"

### Evidência
- [ ] Screenshot do dashboard acessível após restart do servidor
- [ ] Log do servidor mostrando carregamento da chave RSA do ambiente (não geração)

---

## Cenário 3 — Logout limpa o cookie

### Passos
1. Estar logado na aplicação
2. Clicar no botão de logout (menu de usuário)
3. Após logout, abrir DevTools → Application → Cookies
4. Verificar se o cookie `accessToken` ainda existe
5. Tentar acessar uma rota protegida diretamente (ex: `/items`)
6. Verificar se o usuário é redirecionado para o login

### Resultado Esperado
- Cookie `accessToken` removido (ou com `Max-Age=0` / expirado)
- Acesso a rota protegida sem cookie → redirect para `/login`
- Nenhum loop de redirect

### Evidência
- [ ] Screenshot da aba Cookies após logout (sem `accessToken`)

---

## Cenário Negativo 1 — Rate limiting no login

### Passos
1. Na tela de login, executar 11 tentativas de login com senha errada **em sequência rápida** (usar mesma senha inválida)
2. Anotar o código HTTP da 11ª tentativa
3. Verificar o header `Retry-After` na resposta

### Resultado Esperado
- Tentativas 1–10: HTTP 401 (credenciais inválidas)
- Tentativa 11+: HTTP 429 com header `Retry-After` indicando tempo de espera
- Resposta 429 com `Content-Type: application/problem+json`

### Evidência
- [ ] Screenshot das respostas HTTP no DevTools → Network mostrando 429
- [ ] Screenshot do header `Retry-After` presente

---

## Cenário Negativo 2 — Acesso sem autenticação

### Passos
1. Fazer logout (ou limpar cookies manualmente)
2. Tentar acessar `/items` diretamente via URL
3. Verificar comportamento do redirect

### Resultado Esperado
- Redirect imediato para `/login` sem flash de tela branca (middleware de edge — TASK-045)
- URL de retorno preservada (ex: login redirecta de volta para `/items` após autenticação)

### Notas / Riscos
- TASK-001: verificar especificamente nos logs que a chave RSA foi **carregada** do ambiente, não **gerada** — a log line deve dizer algo como "RSA key loaded from environment" não "RSA key generated"
- TASK-002: verificar que o header `Authorization: Bearer ...` **não** está sendo enviado manualmente pelo `apiClient.ts` (inspecionar requests no Network)
- Se SameSite=Strict for usado com redirecionamentos OAuth futuros, pode quebrar o fluxo — anotar como risco
