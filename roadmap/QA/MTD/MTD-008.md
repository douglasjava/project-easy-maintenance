# MTD-008 — Performance Frontend e Cache React Query

## Flow
[FLOW-010](../Flow/FLOW-010.md) — Performance Frontend e Cache React Query

## Objetivo
Validar que o React Query reutiliza dados em cache ao navegar entre páginas (sem re-fetch desnecessário), que skeletons de loading aparecem durante carregamento inicial, e que a autenticação ocorre sem flash de tela branca.

## Pré-condições
- DevTools do Chrome aberto com aba **Network** ativa
- Usuário autenticado com dados já existentes no sistema (itens, manutenções)
- Conexão de internet estável (para testar cache) e capacidade de simular throttling (Slow 3G)

---

## Cenário 1 — Cache React Query: sem re-fetch na segunda visita (< 2 min)

### Passos
1. Abrir DevTools → aba **Network** → clicar em **Clear** para limpar o histórico
2. Acessar a página `/items` pela primeira vez
3. Observar as requisições feitas no painel Network (anotar chamadas para `/items` ou equivalente)
4. Navegar para `/maintenances` (aguardar carregar)
5. Navegar de volta para `/items`
6. Verificar o painel Network — observar se houve nova chamada para `/items`

### Resultado Esperado
- **Primeira visita** a `/items`: chamada feita à API (ex: `GET /items` visível no Network)
- **Segunda visita** a `/items` (dentro de 2 minutos): **zero** novas chamadas para o endpoint de listagem de itens
- Os dados são exibidos imediatamente sem loading

### Evidência
- [ ] Screenshot do Network tab na segunda visita mostrando ausência de request para `/items`

---

## Cenário 2 — Sem re-fetch ao trocar de aba (window focus)

### Passos
1. Estar na página `/items` com dados carregados
2. Abrir uma nova aba do browser e navegar em outro site por 30 segundos
3. Voltar para a aba da aplicação (clicar na aba)
4. Verificar o painel Network — verificar se novas requisições foram disparadas

### Resultado Esperado
- **Nenhuma** nova requisição disparada ao retornar para a aba da aplicação
- Dados continuam exibidos sem loading

### Evidência
- [ ] Screenshot do Network tab sem novas requisições após retornar à aba

---

## Cenário 3 — Skeleton de loading na primeira carga

### Passos
1. Fazer logout (limpar cache e estado de autenticação)
2. No DevTools → Network → selecionar **Slow 3G** em throttling
3. Fazer login e aguardar o redirecionamento para o dashboard
4. Navegar para `/items`
5. Observar o estado da tela durante o carregamento dos dados

### Resultado Esperado
- Durante o carregamento, é exibido um **skeleton** ou estado de loading (não tela em branco)
- Skeleton visível nas rotas: `/` (dashboard), `/items`, `/maintenances`, `/organizations`
- Nenhum layout quebrado durante o estado de loading

### Evidência
- [ ] Screenshot do skeleton de loading em pelo menos 2 rotas diferentes

---

## Cenário 4 — Contagem de requisições em `/items` (sem N+1)

### Passos
1. Limpar o histórico do Network (Clear)
2. Acessar `/items` pela primeira vez (sem cache)
3. Aguardar o carregamento completo da lista
4. Contar o número de requisições feitas para endpoints de dados (excluindo recursos estáticos como CSS/fonts)
5. Verificar se há padrão N+1 (N requests para permissões individuais)

### Resultado Esperado
- No máximo **2 requisições** de dados na carga inicial de `/items`:
  1. `GET /items` (lista de itens)
  2. `GET /permissions/bulk` ou equivalente (permissões em bulk)
- **Zero** requests individuais por item (ex: não deve haver N requests de `/permissions/{itemId}`)

### Evidência
- [ ] Screenshot do Network tab com todas as requisições de dados listadas (máximo 2)

## Cenário 5 — Middleware de autenticação sem flash de tela

### Passos
1. Fazer logout da aplicação
2. Copiar a URL de uma rota protegida (ex: `/items`)
3. Colar a URL diretamente no browser e pressionar Enter
4. Observar o comportamento do redirect

### Resultado Esperado
- Redirect imediato para `/login` **sem flash de tela branca** ou tela do dashboard por alguns milissegundos
- URL de destino pode ser preservada como query param (ex: `/login?redirect=/items`)

### Evidência
- [ ] Gravação de tela (opcional) ou descrição do comportamento observado

---

## Cenário 6 — Expiração do cache após staleTime (2 minutos)

### Passos
1. Acessar `/items` e aguardar carregamento
2. Aguardar **mais de 2 minutos** sem navegar para outra página
3. Navegar para `/maintenances` e voltar para `/items`
4. Verificar se nova requisição foi disparada desta vez

### Resultado Esperado
- Após 2 minutos (staleTime expirado), **nova requisição** é feita ao retornar para `/items`
- Comportamento correto de revalidação de cache

### Evidência
- [ ] Screenshot do Network tab com nova requisição após > 2 minutos

### Notas / Riscos
- Verificar que queries que precisam de dados sempre frescos (ex: billing em tempo real, notificações) têm `staleTime: 0` localmente — não devem ser afetadas pelo global de 2 min
- O middleware de auth (TASK-045) opera no edge — testar em ambiente de produção/staging para comportamento correto, não apenas localmente
