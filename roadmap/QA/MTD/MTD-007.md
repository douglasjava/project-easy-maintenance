# MTD-007 — Soft Delete e Integridade de Histórico

## Flow
[FLOW-009](../Flow/FLOW-009.md) — Soft Delete e Histórico Regulatório

## Objetivo
Verificar que a deleção de entidades críticas não remove fisicamente os dados do banco, preserva o histórico de compliance, e que registros deletados não aparecem nas listagens normais da aplicação.

## Pré-condições
- Acesso ao banco de dados de staging (MySQL client: TablePlus, DBeaver, ou linha de comando)
- Usuário com permissão para criar e deletar itens, manutenções e usuários
- Pelo menos 1 item com 3 manutenções registradas criado previamente

---

## Cenário 1 — Soft delete de item de manutenção

### Passos
1. Anotar o `id` de um item de manutenção existente com manutenções associadas
2. Via UI ou API, deletar o item (botão de exclusão ou `DELETE /items/{id}`)
3. Verificar no frontend: acessar `/items` e confirmar que o item **não aparece** na lista
4. Verificar diretamente no banco:
   ```sql
   SELECT id, name, deleted_at FROM maintenance_items WHERE id = {id_do_item};
   ```
5. Verificar se `deleted_at` está preenchido com um timestamp recente

### Resultado Esperado
- Item **não aparece** na listagem da UI
- Registro **existe no banco** com `deleted_at IS NOT NULL`
- HTTP response da deleção: **200 OK** ou **204 No Content** (não 500)

### Evidência
- [ ] Screenshot da listagem de itens sem o item deletado
- [ ] Screenshot da query SQL com `deleted_at` preenchido

## Cenário 2 — Histórico de manutenções preservado após delete do item

### Passos
1. Antes de deletar o item, anotar os IDs das manutenções associadas
2. Deletar o item (soft delete)
3. Verificar no banco:
   ```sql
   SELECT id, item_id, performed_at, deleted_at 
   FROM maintenances 
   WHERE item_id = {id_do_item};
   ```

### Resultado Esperado
- Manutenções associadas ao item **ainda existem no banco**
- `deleted_at` das manutenções **permanece NULL** (não foram deletadas em cascata)
- Histórico de execuções preservado para auditoria futura

### Evidência
- [ ] Screenshot da query SQL mostrando manutenções com `item_id` do item deletado ainda existindo

## Cenário 3 — Soft delete de usuário preserva histórico

### Passos
1. Criar um usuário de teste com pelo menos 2 manutenções registradas em seu nome
2. Deletar o usuário via UI (como admin) ou via API `DELETE /users/{id}`
3. Verificar no banco:
   ```sql
   SELECT id, name, email, deleted_at FROM users WHERE id = {id_do_usuario};
   ```
4. Verificar manutenções criadas pelo usuário:
   ```sql
   SELECT id, created_by, performed_at FROM maintenances WHERE created_by = {id_do_usuario};
   ```

### Resultado Esperado
- Usuário com `deleted_at IS NOT NULL` no banco
- Manutenções criadas pelo usuário **preservadas** no banco
- Usuário **não aparece** na listagem de usuários da organização

### Evidência
- [ ] Screenshot da query SQL do usuário deletado
- [ ] Screenshot da query SQL das manutenções preservadas

## Cenário 4 — Filtro automático nas listagens

### Passos
1. Criar 3 itens e anotar seus IDs
2. Deletar o item de ID X
3. Fazer `GET /items` via API ou acessar a UI
4. Verificar que apenas 2 itens aparecem (o deletado não)
5. Verificar na response JSON que o ID do item deletado não está presente
6. Confirmar no banco que há 3 registros, mas apenas 2 com `deleted_at IS NULL`

### Resultado Esperado
- API retorna apenas registros com `deleted_at IS NULL`
- Banco tem os 3 registros (1 com `deleted_at` preenchido)
- Contagem da API = contagem SQL com filtro `WHERE deleted_at IS NULL`

### Evidência
- [ ] Screenshot da resposta da API com 2 itens
- [ ] Screenshot da query SQL mostrando 3 registros totais, 2 sem `deleted_at`

## Cenário 5 — Verificação de soft delete em todas as entidades

### Passos
Verificar no banco se as colunas existem em todas as entidades críticas:
```sql
SHOW COLUMNS FROM maintenance_items LIKE 'deleted_at';
SHOW COLUMNS FROM maintenances LIKE 'deleted_at';
SHOW COLUMNS FROM users LIKE 'deleted_at';
SHOW COLUMNS FROM organizations LIKE 'deleted_at';
```

### Resultado Esperado
- Coluna `deleted_at TIMESTAMP NULL` presente em todas as 4 tabelas
