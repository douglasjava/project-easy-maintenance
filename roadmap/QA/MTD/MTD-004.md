# MTD-004 — Onboarding com Cenários de Erro

## Flow
[FLOW-004](../Flow/FLOW-004.md) — Onboarding

## Objetivo
Verificar que erros no fluxo de onboarding exibem mensagens específicas e orientativas (não genéricas), e que falhas de integração (Asaas indisponível) não bloqueiam completamente o usuário.

## Pré-condições
- Ambiente de staging ou local com backend rodando
- Acesso para simular falha do Asaas (desligar o serviço no Docker Compose local, ou forçar circuit breaker a OPEN)
- E-mail de teste disponível (não cadastrado previamente)
- Ferramentas de DevTools ativas

---

## Cenário 1 — Onboarding completo (happy path)

### Passos
1. Acessar a URL de registro/onboarding
2. Preencher todos os campos com dados válidos:
   - Nome completo
   - CPF ou CNPJ válido (usar gerador de CPF/CNPJ para testes)
   - E-mail não cadastrado
   - Senha com os critérios definidos
   - Nome da organização
3. Selecionar um método de pagamento (ou avançar como trial)
4. Submeter o formulário
5. Verificar redirecionamento para o dashboard

### Resultado Esperado
- Usuário criado com sucesso
- Redirecionamento para o dashboard
- Estado de loading visível durante o processamento (spinner, skeleton ou texto "Criando sua conta...")
- Trial ativo visível no dashboard (banner de trial)

### Evidência
- [ ] Screenshot do dashboard após onboarding com banner de trial

---

## Cenário 2 — CPF/CNPJ inválido

### Passos
1. No formulário de onboarding, inserir um CPF inválido: `000.000.000-00`
2. Manter os demais campos com dados válidos
3. Submeter o formulário

### Resultado Esperado
- Mensagem de erro exibida **inline no campo de CPF** (não apenas toast global)
- Mensagem específica, ex: "CPF inválido — verifique os dígitos"
- Formulário **não submetido** enquanto campo inválido

### Evidência
- [ ] Screenshot do campo CPF com mensagem de erro inline

---

## Cenário 3 — E-mail já cadastrado

### Passos
1. No formulário de onboarding, inserir um e-mail que já existe no sistema
2. Preencher os demais campos com dados válidos
3. Submeter o formulário

### Resultado Esperado
- Mensagem de erro específica informando que o e-mail já está em uso
- Orientação sobre como proceder (ex: "Este e-mail já está cadastrado. Tente fazer login.")
- Link para a tela de login disponível na mensagem

### Evidência
- [ ] Screenshot da mensagem de e-mail duplicado com orientação

---

## Cenário 4 — Falha do Asaas (circuit breaker)

### Passos
1. Simular falha do Asaas: desligar o serviço Asaas no ambiente local ou forçar `OPEN` state no circuit breaker
2. Preencher o formulário de onboarding com dados válidos
3. Selecionar método de pagamento e submeter
4. Verificar se o sistema permite prosseguir sem configurar pagamento agora

### Resultado Esperado
- Mensagem clara informando que o serviço de pagamento está temporariamente indisponível
- Opção de **continuar para o trial sem configurar pagamento agora**
- Nenhuma mensagem genérica "Algo deu errado"
- Usuário consegue acessar o produto em modo trial

### Evidência
- [ ] Screenshot da mensagem específica de Asaas indisponível
- [ ] Screenshot do dashboard acessível após fallback para trial

---

## Cenário 5 — Timeout de rede

### Passos
1. No DevTools, ativar Network → throttling → **Offline** por 3 segundos enquanto submete o formulário, depois reativar
2. Verificar a mensagem exibida ao usuário

### Resultado Esperado
- Mensagem informando falha de conectividade
- Botão **"Tentar novamente"** visível na tela
- Dados do formulário **não perdidos** após o erro

### Evidência
- [ ] Screenshot da mensagem de erro de rede com botão de retry

---

## Cenário 6 — Estado de loading durante submissão

### Passos
1. No DevTools, ativar Network throttling → **Slow 3G**
2. Preencher formulário válido e submeter
3. Observar o comportamento durante o processamento

### Resultado Esperado
- Loading state visível (spinner, botão desabilitado, ou skeleton)
- Impossível submeter o formulário duas vezes enquanto a requisição está em andamento
- Nenhum estado de UI quebrado durante o loading

### Evidência
- [ ] Screenshot do formulário em estado de loading

---

## Cenário 7 — Verificação via Sentry

### Passos
1. Completar os cenários acima em produção/staging (onde Sentry está configurado)
2. Acessar o painel do Sentry para o projeto
3. Verificar se algum `console.error()` foi capturado durante os testes de onboarding

### Resultado Esperado
- Zero erros não tratados capturados no Sentry durante o fluxo de onboarding
- Erros esperados (400 de validação) não aparecem no Sentry como exceptions

### Notas / Riscos
- TASK-016 menciona substituir `console.error()` por handler adequado — verificar se Sentry está capturando erros indevidos
- O fallback de Asaas falhando depende do circuit breaker (TASK-008) estar configurado corretamente
