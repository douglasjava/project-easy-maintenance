# MTD-002 — Isolamento Multi-Tenant via X-Org-Id

## Flow
[FLOW-002](../Flow/FLOW-002.md) — Isolamento Multi-Tenant

## Objetivo
Verificar que um usuário autenticado em uma organização não consegue acessar, listar ou modificar dados de outra organização por meio da manipulação manual do header `X-Org-Id`.

## Pré-condições
- Dois usuários criados em organizações distintas: **Org A** e **Org B**
- Cada organização com dados exclusivos: itens, manutenções e usuários diferentes
- Acesso ao Postman, Insomnia ou DevTools com capacidade de interceptar/modificar headers
- Anotar os `orgCode` de cada organização (visível nos requests do frontend autenticado)

---

## Cenário 1 — X-Org-Id de outra organização → 403

### Passos
1. Autenticar como usuário da **Org A** no frontend
2. Abrir DevTools → aba **Network** → identificar uma request para `/items` ou `/maintenances`
3. Copiar os headers da request (especialmente `Cookie` com o token JWT e `X-Org-Id`)
4. Abrir Postman e criar uma request `GET /items` com os mesmos headers
5. Substituir o valor do header `X-Org-Id` pelo `orgCode` da **Org B**
6. Enviar a request
7. Verificar o código HTTP e o body da resposta

### Resultado Esperado
- HTTP **403 Forbidden**
- Body com `ProblemDetail` padronizado (ex: `{"type": "...", "status": 403, "detail": "..."}`)
- **Zero** dados da Org B retornados

### Evidência
- [ ] Screenshot da request no Postman com X-Org-Id da Org B
- [ ] Screenshot da resposta 403 com body ProblemDetail

---

## Cenário 2 — Verificação em múltiplos endpoints

### Passos
Repetir o Cenário 1 para os seguintes endpoints com X-Org-Id da Org B:
1. `GET /items`
2. `GET /maintenances`
3. `GET /users`
4. `POST /items` (body com dados válidos da Org A)
5. `DELETE /items/{id_da_org_A}`

### Resultado Esperado
- **403** em todos os endpoints

### Evidência
- [ ] Lista de endpoints testados com status code de cada um

---

## Cenário 3 — X-Org-Id válido e correto → 200 (regressão)

### Passos
1. Com as mesmas credenciais da Org A e `X-Org-Id` correto da Org A
2. Fazer `GET /items`
3. Verificar resposta

### Resultado Esperado
- HTTP **200 OK**
- Lista de itens da Org A retornada normalmente

### Evidência
- [ ] Screenshot da resposta 200 com dados da Org A

---

## Cenário 4 — Filtro automático de tenant no repositório

### Passos
1. No banco de dados, verificar se existem registros de múltiplos tenants na tabela `maintenance_items`
2. Autenticar como usuário da Org A com `X-Org-Id` correto
3. Fazer `GET /items` e anotar a quantidade de itens retornados
4. Verificar no banco: `SELECT COUNT(*) FROM maintenance_items WHERE org_code = 'ORG_A_CODE'`
5. Confirmar que o count do endpoint = count da query SQL com filtro

### Resultado Esperado
- Endpoint retorna exatamente o número de itens da Org A (não de outras orgs)
- Registros de outros tenants não aparecem na lista

### Evidência
- [ ] Screenshot da query SQL com o count da Org A
- [ ] Screenshot da resposta do endpoint com a mesma quantidade

---

## Cenário Negativo — JWT expirado com X-Org-Id válido

### Passos
1. Aguardar o token JWT expirar (ou manipular o cookie com token expirado)
2. Fazer request para `/items` com `X-Org-Id` válido

### Resultado Esperado
- HTTP **401 Unauthorized** (não 403)
- Usuário redirecionado para login

### Notas / Riscos
- Verificar especificamente se o erro é 403 ou 401 — a semântica importa para segurança
- Endpoint de admin cross-tenant (se existir no produto) deve ter escopo especial validado separadamente
- TASK-017 garante fail-fast no repositório sem TenantContext — verificar que endpoints internos ou jobs que chamam repositórios diretamente também são protegidos
